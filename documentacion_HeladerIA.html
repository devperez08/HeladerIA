Descripción General

HeladerIA es un sistema de gestión integral para una heladería, diseñado para automatizar y digitalizar las operaciones del negocio mediante tecnología moderna de Inteligencia Artificial. El proyecto integra un backend robusto, una interfaz gráfica de usuario interactiva y un asistente inteligente capaz de tomar pedidos, tanto por texto como por voz, de manera completamente offline.

🤖
Asistente IA de Chat
Chatbot que entiende lenguaje natural para tomar pedidos y resolver consultas sobre el menú.
🎙️
Agente de Voz Offline
Reconocimiento de voz en español sin internet (Vosk + Piper). Ciclo completo de audio.
📦
Gestión de Inventario
CRUD completo de sabores con control de stock automático via triggers SQLite.
💰
Registro de Ventas
Registro manual y automático vía IA con historial detallado y transacciones seguras.
🚚
Gestión de Compras
Control de abastecimiento con actualización automática del stock al registrar entradas.
🐳
Docker + Raspberry Pi
Sistema completo contenedorizado con script de despliegue automatizado para Raspberry Pi 4.

El sistema usa SQLite como motor de base de datos, embebida directamente en el proyecto. No requiere instalar ningún servidor de base de datos, lo que la hace perfecta para Raspberry Pi. La base de datos se crea y siembra automáticamente en el primer arranque.

Triggers Automáticos de Stock

⚡
Gracias a los triggers, el stock nunca necesita actualizarse manualmente. Al registrar una venta o compra, la base de datos se actualiza de forma atómica y consistente.
actualizar_stock_venta

Se ejecuta AFTER INSERT ON detalle_ventas → Descuenta la cantidad vendida del stock del sabor correspondiente.

actualizar_stock_compra

Se ejecuta AFTER INSERT ON detalle_compras → Suma la cantidad comprada al stock del sabor correspondiente.

API REST — Backend

El backend está construido con FastAPI y centraliza toda la lógica de negocio. Se ejecuta con Uvicorn en el puerto 8000. La documentación interactiva está disponible en http://localhost:8000/docs.

🏥 Sistema

GET /health Verifica que la API está activa → {"status": "online"}

📦 Inventario

GET /inventario Retorna la lista completa de sabores con precio y stock
POST /inventario Agrega un nuevo sabor — Body: {nombre, precio, stock}
PUT /inventario Actualiza un sabor existente — Body: {id, nombre, precio, stock}
DEL /inventario/{sabor_id} Elimina un sabor (solo si no tiene ventas asociadas)
⚠️
No se puede eliminar un sabor con ventas históricas. La API retorna HTTP 400 con mensaje explicativo.

💰 Ventas

POST /ventas Registra una venta — Body: {"items": [{"idSabor": 1, "cantidad": 2}]}
GET /ventas Retorna historial de ventas con detalles por ítem

🚚 Compras

POST /compras Registra entrada de mercancía — Body: {"items": [...], "total_compra": 0.0}

🤖 Inteligencia Artificial

POST /ai/chat Procesa mensaje de texto y responde — Body: {"message": "quiero 2 de fresa"}
GET /ai/status Consulta si el agente de voz está activo → {"voice_active": false}
POST /ai/voice/toggle Enciende o apaga el agente de voz — Body: {"active": true}

Lógica de Venta con Transacciones

La función realizar_venta_interna() usa transacciones SQLite atómicas que garantizan la integridad de los datos:

def realizar_venta_interna(items):
    conn.execute("BEGIN TRANSACTION")
    try:
        # 1. Verifica stock de cada ítem
        # 2. Inserta en tabla 'ventas'
        # 3. Inserta en 'detalle_ventas'
        # 4. Trigger actualiza stock automáticamente
        conn.commit()
    except:
        conn.rollback()  # Si algo falla, se revierte todo
        raise

Módulo de Inteligencia Artificial

El módulo de IA implementa una arquitectura híbrida: detección determinista en Python puro para la lógica crítica de negocio, con el LLM como respaldo conversacional.

Flujo de Procesamiento del Chat

Recibir mensaje del usuario y consultar el catálogo fresco desde la BD.
Detección Python puro (detectar_pedido): analiza intención de compra, cantidad y sabor sin depender del LLM.
Si es un pedido válido: verificar stock → ejecutar realizar_venta_interna() → retornar confirmación con total y número de venta.
Si el sabor no existe: listar todos los sabores disponibles amigablemente.
Si es una consulta: enviar al LLM (qwen2.5:0.5b via Ollama) con prompt minimalista → respuesta conversacional en 1 frase.

Detección de Intención (Python Puro)

TipoPalabras clave
Preguntas / Consultas"qué", "cómo", "hay", "precio", "cuánto", "disponible", "menú", "lista"
Intención de compra"quiero", "dame", "quisiera", "pido", "nécdote", "tráeme", "véndeme", "compro"
Cantidades (palabras)"un", "dos", "tres", "cuatro", "cinco"... hasta "cien"
Cantidades (dígitos)Regex \b\d+\b — detecta cualquier número

Uso del LLM con Ollama

# Para consultas conversacionales (respuesta breve)
respuesta = ollama.chat(
    model="qwen2.5:0.5b",
    messages=[{'role': 'user', 'content': prompt}]
)

# Para extracción estructurada (pedidos de voz)
respuesta = ollama.chat(
    model="qwen2.5:0.5b",
    messages=[...],
    format="json"  # Fuerza respuesta JSON estricta
)
# Resultado: {"idSabor": 3, "cantidad": 2}

Agente de Voz Offline

El agente de voz (Local_agent_voz.py) implementa un ciclo completo de interacción por voz, 100% offline, sin necesidad de internet.

Pipeline de Procesamiento

🎤
Micrófono
PyAudio 16kHz PCM
→
👂
Vosk STT
Reconocimiento offline ES
→
🧠
Ollama LLM
Extrae JSON del pedido
→
⚙️
FastAPI
POST /ventas
→
🔊
Piper TTS
Genera audio .wav
→
🔈
Altavoz
aplay ALSA

Componentes Principales

1. escuchar_cliente() — Captura de Audio

  • PyAudio abre el micrófono en formato PCM 16-bit mono a 16,000 Hz
  • Procesa en chunks de 4,000 frames para baja latencia
  • Muestra resultados parciales en tiempo real mientras el usuario habla
  • Se detiene cuando Vosk entrega un resultado final (AcceptWaveform → True)

2. decodificar_y_vender() — Procesamiento IA

  • Envía el texto reconocido a Ollama con el catálogo actual
  • Fuerza respuesta JSON: {"idSabor": X, "cantidad": Y}
  • Llama a POST /ventas si el JSON es válido
  • Maneja errores: sabor inexistente, stock insuficiente, error de red

3. hablar() — Síntesis de Voz

echo "Texto a decir" | ./piper/piper --model voz_es.onnx --output_file salida.wav
aplay -q salida.wav

Agente por Texto (modo debug)

Local_agent_voz_texto.py reemplaza el micrófono con input() de teclado. Útil para probar el pipeline de IA sin necesitar hardware de audio.

Flujo de Trabajo Completo

Escenario 1: Venta Manual desde el Dashboard

Operador abre el Dashboard en http://localhost:8501
Navega a Ventas → pestaña "Nueva Venta"
Selecciona cantidad de cada sabor — el total se calcula automáticamente en tiempo real
Presiona "Finalizar Venta"
Frontend llama a POST /ventas con los ítems
FastAPI ejecuta realizar_venta_interna() en transacción SQLite atómica
Trigger descuenta stock automáticamente → Dashboard muestra animación de globos ✅

Escenario 2: Pedido por Chat de IA

Usuario escribe en el chat: "quiero 3 de fresa"
Frontend envía POST /ai/chat {"message": "quiero 3 de fresa"}
detectar_pedido() identifica: intención "quiero" ✓ · sabor "Fresa" (ID 3) ✓ · cantidad 3 ✓
Se ejecuta realizar_venta_interna() (misma lógica que venta manual)
Chat responde: "✅ ¡Pedido registrado! 3x Fresa | Total: $10,500 · Venta #15"

Escenario 3: Pedido por Voz (Agente Offline)

Operador activa el toggle "Activar Micro" en el sidebar del dashboard
API lanza Local_agent_voz.py como subproceso → Piper saluda al cliente por el altavoz
Cliente habla al micrófono: "dame dos de chocolate"
Vosk transcribe a texto → Ollama extrae {"idSabor": 2, "cantidad": 2}
Agente llama a POST /ventas → API registra la venta y actualiza stock
Piper sintetiza respuesta → altavoz reproduce: "¡Excelente elección! Pedido procesado."
El ciclo vuelve a escuchar en el paso 3

Escenario 4: Reabastecimiento de Inventario

Operador navega a Compras → "Registrar Compra"
Ingresa la cantidad a comprar por sabor → el sistema calcula la inversión total
Presiona "Registrar Entrada" → frontend llama a POST /compras
Trigger SQLite suma las cantidades al stock → inventario actualizado instantáneamente ✅

Despliegue con Docker

El sistema está contenedorizado con tres servicios orquestados por Docker Compose.

ServicioPuertoDescripciónNotas
api:8000FastAPI + Uvicorn (backend)Volumen persistente para la BD
app:8501Dashboard Streamlit (frontend)Depende del servicio api
voz—Agente de Voz offlineAcceso a /dev/snd, network_mode: host
ℹ️
El servicio voz usa network_mode: host para acceder correctamente al servidor de audio PulseAudio de la Raspberry Pi. Los modelos pesados se montan como volúmenes para no inflar la imagen Docker.

Comandos Esenciales

# Construir y levantar todo
docker-compose up --build -d

# Ver logs en tiempo real
docker-compose logs -f

# Ver logs solo del agente de voz
docker-compose logs -f voz

# Detener todos los servicios
docker-compose down

# Reiniciar un servicio específico
docker-compose restart api

El script deploy_raspberry.sh automatiza el despliegue completo en una Raspberry Pi 4 con un único comando:

bash deploy_raspberry.sh
